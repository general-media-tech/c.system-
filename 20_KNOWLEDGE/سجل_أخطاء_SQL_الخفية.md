## 🔴 SQL-1 — دالة العدّاد معرّفة مرتين بنوعين متعارضين (الخطأ الذي واجهته)

**الخطأ الظاهر:**
```
ERROR: 42P13: cannot change return type of existing function
HINT: Use DROP FUNCTION increment_settings_counter(text) first.
```

**السبب الجذري:** `increment_settings_counter` معرّفة **مرتين**:

| الملف | يُرجع | العمود المستعمل |
|---|---|---|
| `GMT_MASTER_SCHEMA:25` | `bigint` | `counter` (رقمي) |
| `GMT_ORDERS_FIXES:10` | `integer` | `value` (نصّي) + قفل `for update` |

**الأخطر من الخطأ نفسه:** الكود الفعلي (`05_الأوردرات.html:873,881,888`) **يقرأ/يكتب من عمود `value`** — أي أن **نسخة ORDERS_FIXES هي الصحيحة**، ونسخة MASTER خاطئة (تكتب بعمود لا يقرأه الكود).

**الأثر لو شُغّلت النسخة الخاطئة:** عدّاد أرقام الأوردرات يكتب بمكان والكود يقرأ من مكان آخر ⇒ **أرقام أوردرات مكررة أو مفقودة بصمت**.

**الحل اليدوي الفوري (نُفّذ مرة واحدة):**
```sql
DROP FUNCTION IF EXISTS increment_settings_counter(text);
-- ثم شغّل نسخة ORDERS_FIXES فقط (value + for update)، وتخطّى نسخة MASTER (سطر 25-39)
```

**النوع:** تعارض تعريف مزدوج + خطأ ترتيب. **الخطورة:** 🔴 Critical (يكسر ترقيم الأوردرات).

## 🔴 SQL-2 — audit_log و الدالة الصحيحة خارج ترتيب النشر الموثّق

**المشكلة:** الـREADME الرئيسي يوصي بترتيب:
```
1. GMT_MASTER_SCHEMA  2. GMT_BOTS  3. GMT_POS_FIXES  4. GMT_WARRANTY
```
لكن **ملفّين حرجين غير مذكورين** بهذا الترتيب:
- `GMT_ACCOUNTING_GUARDS.txt` — **ينشئ جدول `audit_log`** (المساءلة).
- `GMT_ORDERS_FIXES.txt` — **يعرّف النسخة الصحيحة** من دالة العدّاد + عمود `tg_msg_id`.

**الأثر لو اتُّبع الـREADME حرفياً:**
1. `MASTER_SCHEMA` يحاول تفعيل RLS على `audit_log` — لكن الجدول غير موجود بعد (لأن ACCOUNTING_GUARDS لم يُشغّل) ⇒ الحلقة **تتخطّاه بصمت** (`if exists`) ⇒ **`audit_log` يعمل بلا حماية RLS** لو أُنشئ لاحقاً.
2. دالة العدّاد تبقى بنسخة MASTER الخاطئة ⇒ SQL-1 يتكرر.

**الخطورة:** 🔴 Critical (المساءلة بلا حماية + العدّاد مكسور). **النوع:** نقص بترتيب النشر الموثّق.

**الترتيب الصحيح الكامل (للتوثيق — لا تُنفّذ الآن):**
```
DB1_MAIN بالترتيب:
  1. GMT_ACCOUNTING_GUARDS.txt   (ينشئ audit_log · stock_moves · period_locks أولاً)
  2. GMT_MASTER_SCHEMA           (يفعّل RLS — الآن audit_log موجود)
  3. GMT_ORDERS_FIXES.txt        (النسخة الصحيحة للعدّاد + tg_msg_id)
     ⚠️ سبقها بـ: DROP FUNCTION IF EXISTS increment_settings_counter(text);
  4. GMT_POS_FIXES.txt
  5. GMT_BOTS
  باقي ملفات .txt القديمة (كلها if not exists — آمنة)
DB3_WARRANTY: GMT_WARRANTY_TRACKING
```

## 🔍 كيف أكتشف المالك أخطاء مماثلة مستقبلاً

هذه الفئة (تعارضات SQL) لا يكشفها إلا التشغيل. **الفاحص الذاتي 🧪 يكشف بعضها** (يفحص وجود الأعمدة والجداول)، لكن لا يكشف تعارض تعريف الدوال. التوصية: تشغيل SQL على قاعدة تجريبية أولاً دائماً.

**الخطورة الإجمالية لهذه الفئة:** 🔴 — لأنها صامتة وتمسّ الترقيم والمساءلة، وهما من أهمّ ضمانات النظام.

## 🔴 SQL-3 — تعارض audit_log و period_locks (أُصلح 2026-07-17)

**الخطأ الظاهر:** `ERROR: 42703: column "table_name" does not exist`

**السبب الجذري:** جدولان (`audit_log` و `period_locks`) ودالة (`gmt_is_locked`) كانوا **معرّفين بملفّين بأعمدة/توقيعات مختلفة**:

| العنصر | ACCOUNTING_GUARDS (يُشغّل أولاً) | MASTER (ثانياً) |
|---|---|---|
| audit_log | entity · entity_id · before_val · after_val · note · source | table_name · row_id · before_data · after_data · reason |
| period_locks | branch_key · locked_upto · locked_at · note | locked_to · reason |
| gmt_is_locked | (p_branch, p_date) | (p_date) |

بما أن ACCOUNTING يُشغّل أولاً ويُنشئ الجداول بأعمدته، ثم MASTER يجدها موجودة (`if not exists`) لكن يحاول عمل index/دالة على أعمدة غير موجودة ⇒ الخطأ.

**الإصلاح:** أُزيلت التعريفات المكرّرة من MASTER نهائياً. **مصدر واحد لكل جدول = ACCOUNTING_GUARDS فقط.** نفس مبدأ إصلاح SQL-1.

**الدرس:** التعارض لا يظهر بفحص الكود العادي (الجدولان بنفس الاسم)، فقط عند التشغيل. الحل الجذري: مصدر واحد لكل تعريف.

## 🔴 SQL-4/5/6 — أسماء جداول خاطئة أو غير معرّفة (أُصلحت 2026-07-17)

**الأخطاء الظاهرة:** `relation "public.orders" does not exist` (وأخطاء مشابهة محتملة).

**الأسباب والإصلاحات:**

| الرمز | المشكلة | الإصلاح |
|---|---|---|
| SQL-4 | `GMT_ORDERS_TG_MSGID.txt` يستعمل `public.orders` بينما الجدول الحقيقي `gmt_orders` | تصحيح كل المراجع إلى `gmt_orders` |
| SQL-5 | `GMT_SETTLEMENTS.txt` يعمل `alter table branch_transfers` دون تعريف الجدول أولاً | أُضيف `create table if not exists branch_transfers` بكل أعمدته قبل الـalter |
| SQL-6 | `GMT_WARRANTY_TRACKING` ينشئ index على `shipments` دون التأكد من وجوده | الفهرس صار مشروطاً بوجود الجدول (do block) |

**الفحص الشامل:** بعد الإصلاح، فُحصت **كل** ملفات SQL — لا `alter`/`index` على أي جدول غير معرّف، ولا اسم جدول خاطئ.

**الدرس:** أسماء الجداول يجب أن تطابق ما يستعمله الكود (`gmt_` بادئة)، وأي `alter` يحتاج الجدول معرّفاً أو `if exists`.

## 🔴 SQL-7 — الجداول الأساسية غير معرّفة (الأخطر · أُصلح 2026-07-17)

**المشكلة الجذرية:** 16 جدولاً أساسياً (products · invoices · gmt_users · invoice_commissions · invoice_items · gmt_orders · import_log · inv_columns · وغيرها) **يستعملها الكود بكثافة لكن لا يعرّفها أي ملف SQL** بالحزمة. كل الملفات كانت تفترض وجودها مسبقاً (`alter table if exists`).

**السبب:** هذه الجداول أُنشئت بملف SQL قديم عند المالك (غير موجود بالحزمة). فالنشر على قاعدة **جديدة فارغة** كان يفشل — لا توجد جداول أساساً.

**الإصلاح:** أُنشئ ملف `GMT_00_BASE_TABLES.sql` يعرّف كل الجداول الأساسية الـ16 (بأعمدة مستخرجة من استعمال الكود الفعلي). يُشغّل **أولاً قبل كل شيء**. آمن للتكرار (`if not exists`) — لا يمسّ قواعد موجودة.

**+ SQL-8:** `gmt-health.js` كان يستعمل `orders` بدل `gmt_orders` — صُحّح.

**الفحص الشامل النهائي:** كل جدول يستعمله الكود (31 جدولاً) صار معرّفاً أو مغطّى. لا جدول غير معرّف · لا اسم خاطئ · لا تعارض.

**هذا كان سبب معظم أخطاء «does not exist» عند النشر النظيف.**

## 🔴 SQL-9..12 — الفحص الجذري الشامل (كل القواعد · 2026-07-17)

بعد سؤال المالك «ليش تتكرر الأخطاء؟»، أُجري **فحص جذري** يقارن كل ما يستعمله الكود مقابل كل ما تعرّفه الـSQL — عبر **كل القواعد الأربعة**، لا الرئيسية فقط:

| الرمز | المشكلة | الإصلاح |
|---|---|---|
| SQL-9 | 3 دوال RPC مخزنية مفقودة (add_branch_stock · deduct_branch_stock · next_stock_transfer_number) | عُرّفت بملف التأسيس |
| SQL-10 | جدول gmt_telemetry (التتبّع) غير معرّف | عُرّف بملف التأسيس |
| SQL-11 | قاعدة الكفالة (DB3): warranties · activation_logs · shipments غير معرّفة | ملف GMT_00_BASE_WARRANTY.sql |
| SQL-12 | قاعدة الموقع (DB4): site_settings غير معرّف | أُضيف لـGMT_SITE_AGENTS |

**المنهجية الجديدة (جذرية بدل رد الفعل):**
- كل **جدول** يستعمله الكود (rest/v1) → معرّف؟
- كل **دالة RPC** يناديها الكود → معرّفة؟
- عبر **كل القواعد** (رئيسية · كفالة · موقع · متجر)، لا الرئيسية فقط.

**النتيجة النهائية:** 33 جدول + 5 دوال RPC يستعملها الكود — **كلها معرّفة الآن**. صفر ناقص عبر كل القواعد.

**لماذا تكررت الأخطاء سابقاً؟** الفحوص الأولى كانت **صياغية** (هل الكود مكتوب صح؟) لا **ترابطية** (هل يجد الكود ما يحتاجه بالقاعدة؟). والمشكلة لا تظهر إلا على قاعدة نظيفة. الفحص الترابطي الشام012 (هذا) يمسك كل النواقص دفعة واحdة بدل ظهورها فرادى.
